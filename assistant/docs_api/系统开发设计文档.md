# 知识库问答助手 - 系统开发设计文档

## 1. 架构概览

本系统是一个基于 Retrieval-Augmented Generation (RAG) 架构的本地知识库问答助手。
- **服务端架构**：采用 Python + FastAPI 提供轻量级的 HTTP 接口与 SSE 流式响应。
- **本地大模型**：基于 Ollama 部署 Qwen2.5:7b，确保业务数据的绝对隐私和离线环境下的可用性。
- **向量检索引擎**：采用 ChromaDB 作为持久化检索引擎。
- **语义模型选型**：
  - Embedding 模型：`BAAI/bge-small-zh-v1.5`
  - Reranker 重排模型：`BAAI/bge-reranker-base`

## 2. 核心模块与业务流程设计

### 2.1 文档入库与解析 (Ingest Pipeline)
实现在 `ingest.py` 与 `utils/doc_parser.py` 中。
- **多模态文本解析**：支持解析 `.txt`, `.md`, `.docx`, 以及 `.xlsx` 等主流排版文件。对 Excel 表格进行去空行/空列降噪处理（支持长表头截断），对纯树状或 JSON 配置文件能转化为语义性的 Markdown 层级标识。
- **智能分块 (Chunking)**：以 `chunk_size = 500` 和 `chunk_overlap = 100` 为基准的递归切分算法，兼顾碎片上下文连贯与 Token 容量。
- **元数据管理 (Metadata)**：每个文本块独立绑定其来源名 (`source`)、归属目录类型 (`category`) 和编号。其中 Chunk ID 的生成规则包含 `category` 和 `source`，有效避免不同目录下同名文件的冲突。

### 2.2 上下文多轮处理与意图路由 (Query Processing & Routing)
- **对话改写 (Query Rewrite)**：大模型基于近 3 轮的对话历史扮演“问题重写助手”，对用户的隐式代词（如“这个菜单”、“它是怎么计算的”）进行显式上下文意图补全，提升独立检索有效性。
- **软路由与旁路过滤策略 (Soft-Routing & Filter)**：利用正则与特征匹配，将用户提问映射至以下多个垂域：
  1. `chitchat`：闲聊模式，跳过深度检索模块。
  2. `global_menu`：触发旁路兜底查询。直接过滤提取向量库中所有 `category="routes"` 的核心架构或总览节点以全景形态交给模型归纳，避免 Top-K 文本碎片的遗漏。
  3. `route_qa` / `manual_qa` / `requirement_qa`：针对此三个明确域，首先注入 `where={"category": ...}` 进行针对性检索（Top 20）。如该域下无结果，则 fallback 取消过滤重新全库查询。极大地避免了例如“业务操作说明”干涉“页面路由”的情况。
  4. `general_rag`：普通全库查找流程。

### 2.3 深度精排与呈现 (Rerank & Generation)
- **双塔精排策略**：初步使用 Embedding 根据余弦相似度拉取候选池（召回 20 条）以保障覆盖率。为弥合向量相似度和语义相关性的差异，调用 Cross-Encoder 模型双向计算，剔除并缩圈选取匹配度最高的 5 条文档段落。
- **Prompt 工程与约束**：针对软路由得到的每种意图施加针对性强力系统指令：
  - 如路由问题强制模型必须提取并声明页面路径、权限与入口归属，并严禁未命中知识点情况下的自行捏造和强行填补。
  - 操作说明重点凸显分布执行规范。
  - 需求问答侧重分析背后的口径与逻辑。
- **流式追踪与推流**：生成请求体时携带依据的来源信息数组 `sources` 作为前置 Meta 事件下发给前端，随后结合 `requests.post` 与生成器的流式返回能力，实现平滑的“打字机”实时输出体验。

## 3. 部署与后续优化点说明

- 由于接入了 BGE Embedding 与 Reranker，需确保部署环境中有相应的 PyTorch 环境以及网络许可（如 HF Hub 代理配置），这已由 `HF_ENDPOINT` 保障。
- 后续规划：在系统扩大语料库时，若命中逻辑变得更复杂，可考虑将现有的硬规则关键字软路由升级为由轻量级分类模型 (如基于小参数量适配训练或 LLM Few-Shot) 执行意图路由判断。
